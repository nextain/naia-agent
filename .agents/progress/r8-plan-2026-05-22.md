## 2. R8 의 진짜 작업

**naia = 흡수체로서 우리 stack** — pi + Hermes 의 좋은 알고리즘 흡수 + 우리 고유 자산 (bi-temporal, contradiction filter dual-process, R-Tree, sqlite-vec) 유지.

| 출처 | 흡수 대상 |
|---|---|
| **pi** (substrate, MIT) | anchored iterative LLM compaction (`findCutPoint` + `SUMMARIZATION_PROMPT` 5-section) |
| **Hermes** (가장 핫, MIT) | 5-step 구조 (Prune → Protect head → Protect tail → Summarize middle → Iterate) |
| **우리 고유** | naia-memory 의 bi-temporal, contradiction filter dual-process, R-Tree, sqlite-vec, hot/cold tier, decay/importance |

→ **단일 stack `naia`** 가 측정 대상. 분리 측정 (pi-only / pi+naia 등) 아님.

## 3. Ship 전략

사용자 directive: "튜닝하다보면 pi 를 지켜주기 어려울것 같은데"

→ pi 의 코드/인터페이스에 종속하지 않음. pi-mono npm 의존성 X.

| 옵션 | 채택 여부 |
|---|:---:|
| A. pi extension 으로 wire (pi-mono 의존) | ✗ — 튜닝 시 깨질 위험 |
| B. naia 흡수체 (pi 알고리즘 차용 + 우리 인터페이스) | ✓ |
| C. pi 그대로 ship → 점진 진화 | ✗ — 미래 분리 비용 큼 |

→ **B 채택**. pi 의 알고리즘만 차용 (License MIT 인용), 우리 인터페이스 (`prepareCompact` hook).

## 7. ship 기준 (재명시)

naia-os 에 compaction ship 하려면 ≥3 개 조건:

1. **선택 strategy (`naia` 흡수체) 가 평가 framework 의 recap-only 컬럼에서 baseline (off) 이상**
2. **`recapNoOp = 0` fixture** (실제로 압축 동작 + fact 보존)
3. **적대 audit 결과 HALT 0 / MAJOR ≤2** (해결책 명시)

→ 객관 검증 없이 ship 안 함.
