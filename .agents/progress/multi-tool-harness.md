## 1. 디자인 결정

### 1.1 Mirror 패턴 선택 (vs thin pointer)

| 옵션 | 장단 | 결정 |
|---|---|---|
| Thin pointer (`See AGENTS.md`) | 1곳 유지보수, 도구가 follow 안 할 위험 | ❌ |
| **Mirror (동일 콘텐츠 복사)** | 1패스 컨텍스트 로딩, 동기화 자동화로 부담 해결 | ✅ 채택 (naia-os 본보기) |
| Symlink | 깔끔하지만 Windows 호환성 + git 처리 복잡 | ❌ |

**근거**: 도구가 자기 wrapper(`CLAUDE.md` 등)를 우선 읽고 다른 파일을 자동 follow하지 않을 수 있음. mirror는 이 위험을 0으로 만들고, sync script 한 줄로 동기화.

### 1.2 도구별 wrapper 파일 4개

| 파일 | 도구 | 인식 우선도 |
|---|---|---|
| `AGENTS.md` (canonical) | Codex / opencode / Aider 등 | AGENTS.md 직접 읽음 |
| `CLAUDE.md` | Claude Code | CLAUDE.md 우선, AGENTS.md fallback |
| `GEMINI.md` | Gemini CLI | GEMINI.md 우선 또는 `.gemini/settings.json`에서 지정 |
| `OPENCODE.md` | opencode (sst) | AGENTS.md 직접 읽음 (mirror는 explicitness 위해) |
| `CODEX.md` | Codex (OpenAI) | AGENTS.md 직접 읽음 (mirror는 explicitness 위해) |

**naia 자체 도구**: 우리가 만들 `naia` 도구는 AAIF 표준 따라 AGENTS.md 직접 읽도록 설계. **NAIA.md는 만들지 않음**.

### 1.3 도구 전용 행동은 `.{tool}/` 디렉터리에

- `.claude/` — Claude Code hooks, slash commands, skills
- `.gemini/settings.json` — Gemini CLI 설정 (AGENTS.md 경로 지정 등)
- `.opencode/` — opencode 전용 config
- `.codex/` — Codex 전용 config (필요 시)

**Mirror에 도구 전용 행동을 넣지 말 것**. Mirror = canonical SoT 동일 복사. 도구별 hook/command/skill은 `.{tool}/`에.

## 2. Sync 메커니즘

### 2.1 스크립트

`scripts/sync-harness-mirrors.sh`

```bash
# Sync (mirrors 모두 재생성)
scripts/sync-harness-mirrors.sh

# Check (CI / pre-commit, 차이나면 exit 1)
scripts/sync-harness-mirrors.sh --check
```

**동작**:
1. `AGENTS.md` 읽음
2. 각 mirror 앞에 banner (HTML comment) 붙임 — Markdown 렌더링에서 안 보이지만 LLM/grep은 읽음
3. 4개 mirror 파일 덮어쓰기

### 2.2 Banner 예시 (CLAUDE.md 첫 부분)

```html
<!--
⚠️  AUTO-GENERATED MIRROR of AGENTS.md.
Do not edit this file directly. Edit AGENTS.md and run:
  scripts/sync-harness-mirrors.sh

This mirror exists so CLAUDE.md-aware tools (which read this filename
preferentially) get the same Single Source of Truth as AGENTS.md.

Standard: AAIF / agents.md (https://agents.md/)
-->
```

### 2.3 Pre-commit hook (권장)

```bash
ln -sf ../../scripts/sync-harness-mirrors.sh .git/hooks/pre-commit
```

매 commit 시 자동 sync. AGENTS.md 편집 후 `git add AGENTS.md` 하면 4개 mirror도 자동 갱신·stage.

### 2.4 CI 검증 (R2 신설 예정)

`.github/workflows/ci.yml`에 추가:

```yaml
- name: Verify harness mirrors in sync
  run: scripts/sync-harness-mirrors.sh --check
```

PR에서 mirror 누락된 변경 잡힘.

## 3. 일관성 보장 흐름

```
사람/에이전트가 AGENTS.md 편집
            ↓
   pre-commit hook (sync)
            ↓
4개 mirror 자동 재생성 + stage
            ↓
        git commit
            ↓
   CI --check (push 시)
            ↓
   ✅ 모든 mirror 동기 확인
```

이 흐름에서 mirror가 AGENTS.md와 어긋날 수 있는 지점 = 0.

---

## 4. 충돌 시 우선순위

```
.agents/context/agents-rules.json  (machine-readable, 가장 정밀)
        >
AGENTS.md  (사람 + AI 공통 SoT)
        >
.agents/context/project-index.yaml  (인덱스/네비게이션)
        >
mirror (CLAUDE.md / GEMINI.md / OPENCODE.md / CODEX.md)
```

Mirror는 항상 AGENTS.md 자동 동기화이므로 본질적으로 동일. 어긋나면 sync script로 해결.

도구 전용 디렉터리(`.claude/`, `.gemini/` 등)의 설정은 **그 도구의 행동 범위 내에서만** AGENTS.md를 override 가능. forbidden_actions은 override 불가.

## 5. 도구 추가 절차 (새 LLM 도구 등장 시)

예: Anthropic이 새 도구 `Claudia`를 출시 → `CLAUDIA.md` 인식

1. **표준 확인**: 새 도구가 AAIF/agents.md 표준 따르는지 확인 (직접 AGENTS.md 읽으면 mirror 불필요)
2. **mirror 추가 결정**: 도구가 자체 file을 우선 읽는다면 mirror 추가
3. **`scripts/sync-harness-mirrors.sh`** `MIRRORS=(...)` 배열에 `CLAUDIA.md` 추가
4. **`AGENTS.md`** §"Multi-tool harness" 섹션에 새 mirror 명시
5. **`.agents/context/agents-rules.json`** `harness_compatibility.mirrors`에 entry 추가
6. **본 문서** §1.2 표에 추가
7. **sync 실행** + commit

표준 따르는 도구가 등장하면 mirror 불필요 — naia처럼 AGENTS.md 직접 읽도록.

---

## 6. 도구 제거 절차

도구가 deprecated 또는 우리가 안 쓰기로 결정 → 해당 mirror 제거

1. `MIRRORS=(...)`에서 제거
2. mirror 파일 삭제 (`git rm CLAUDIA.md`)
3. `agents-rules.json`에서 제거
4. 본 문서 §1.2 업데이트
5. commit

## 8. 자주 묻는 질문

**Q. mirror가 AGENTS.md와 다르면 어떻게 되나?**
A. pre-commit hook 또는 CI `--check`가 잡음. 머지 차단.

**Q. 도구가 자기 wrapper만 보고 AGENTS.md 안 읽는다면?**
A. mirror가 AGENTS.md와 100% 동일하므로 결과 동일. mirror가 SoT 역할.

**Q. naia 자체 도구는 왜 NAIA.md 없나?**
A. 우리가 만드는 도구라서 AAIF 표준(AGENTS.md 직접 읽음) 따르도록 설계. 추가 wrapper 불필요.

**Q. tool-specific behavior(예: Claude의 hooks)는?**
A. `.claude/`, `.gemini/`, `.opencode/`, `.codex/` 디렉터리에. mirror에 넣지 말 것.

**Q. mirror banner가 LLM 컨텍스트 윈도우를 잡아먹지 않나?**
A. 9줄 (~150 토큰) 이하. 컨텍스트 비용 미미. 대신 도구가 mirror가 auto-generated인 줄 알고 다른 도구로 전환할 때 일관성 신뢰 가능.

**Q. AAIF 표준이 바뀌면?**
A. AGENTS.md만 갱신, sync script 실행. 본 multi-tool-harness.md도 §1.1 standard 링크 갱신.
