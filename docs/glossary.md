# 용어 사전 (naia-agent)

낯선 용어가 나오면 여기서 한 줄로 확인하세요. (개발자·기여자용 — 더 깊은 설명은 [`ARCHITECTURE.md`](ARCHITECTURE.md))

- **헤드리스(headless)** — 화면(GUI)이 없고, 다른 프로그램이 뒤에서 호출해 쓰는 처리 엔진. naia-agent 는 naia-os 가 띄워 쓴다.
- **spawn(스폰)** — 다른 프로그램(여기선 외부 코딩 AI)을 자식 프로세스로 띄우는 것.
- **sub-agent(서브 에이전트)** — naia-agent 가 spawn 해서 실제 작업을 시키는 외부 코딩 AI(pi · opencode · claude-code 등).
- **오케스트레이션(orchestration)** — 여러 sub-agent 를 띄우고 감독해 한 작업을 시키고, 결과를 모아 보고하는 것.
- **provider(프로바이더)** — 실제 답을 생성하는 LLM 서비스(예: ollama, openai 호환, 게이트웨이).
- **wire / wire 스트림** — naia-os 와 naia-agent 가 gRPC 로 주고받는 메시지 흐름(텍스트·도구호출·사용량 등).
- **gRPC** — 프로그램끼리 함수 호출하듯 통신하는 표준 프로토콜. naia-os ↔ naia-agent 경계가 이걸로 연결된다.
- **헥사고날(ports & adapters)** — 핵심 로직은 바깥세상(파일·네트워크)을 모르게 두고, "포트"(경계 인터페이스)와 "어댑터"(그 구현)로만 연결하는 구조.
- **포트(port) / 어댑터(adapter)** — 포트 = 무엇을 할지 정한 인터페이스, 어댑터 = 그걸 실제로 하는 구현(gRPC·provider·memory 등).
- **전송 방식 독립(transport 직교, orthogonal)** — 핵심 로직이 전송 방식(gRPC인지 stdio인지)을 몰라서, 코드 변경 없이 전송만 갈아끼울 수 있다는 뜻.
- **domain / app / ports / adapters / composition** — 프로젝트 내부 폴더(레이어) 이름. domain=순수 규칙, app=처리 흐름, ports=경계, adapters=구현, composition=이들을 연결(배선)하는 곳.
- **배선(wiring) / 배선대기** — 만들어 둔 부품을 실제 진입점에 "연결"하는 것. 배선대기 = 로직·테스트는 끝났지만 호스트가 아직 호출 안 함.
- **dormant(잠자는)** — 어댑터·테스트는 있으나 진입점에 연결 안 된 기능(설계상 다른 컴포넌트가 그 환경을 소유).
- **never-throws** — 어떤 실패(오류·타임아웃·도구부재)에도 예외를 던지지 않고 "실패"를 구조화된 결과로 돌려주는 계약.
- **정직보고(honest reporting)** — 작업 후 실제 변경·검증 결과를 꾸미지 않은 숫자(파일 수·통과/실패)로 보고하는 것.
- **NDJSON** — 한 줄에 JSON 하나씩 줄단위로 흘리는 스트림 포맷(외부 코딩 AI 출력 파싱에 쓰임).
- **계약테스트(contract test)** — 포트(인터페이스)가 약속한 동작을 지키는지 검증하는 테스트.
- **V모델 / 계약 우선** — 코드 전에 시나리오 → 요구사항 → 테스트를 먼저 적고, 코드를 거기에 맞춰 검증하는 개발 방식. 추적 레지스트리 = `docs/progress/01~05`.
- **하네스(harness)** — 절차 문서 + 체크리스트 + 자동 점검 스크립트 묶음. 사람이 매번 기억하지 않아도 품질을 지키게 한다.
