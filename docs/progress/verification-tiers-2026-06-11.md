# 검증 티어링 정책 (속도 ↑, 2026-06-11 루크 지시)

> 문제: 리뷰·검증 시간이 실제 개발을 압도(이번 세션 실측). 원인 = 전 표면에 2-AI 2-clean + 절대-NONE 추격 + z.ai(GLM-5.1) 큰 리뷰 행. 
> 해법: **리뷰 깊이를 리스크·복잡도로 티어링 + 빠른 모델 + 진짜버그에서 정지 + z.ai를 critical path 제거.**

## 근본 원리 (루크): 직교 = 앵커 → 빠른 모델로 충분
이식이 **직교**(계약 = 경계 앵커, 테스트 = 검증 앵커, 코어 순수·주입)되어 있으므로 **drift·버그를 잡는 건 모델의 raw 파워가 아니라 구조(계약+테스트)**다. 따라서:
- **기본 리뷰어 = 빠른 모델**(Haiku 서브에이전트). 무거운 모델(codex gpt-5.5 / GLM-5.1)은 T2 보안핵심에만.
- **테스트가 1차 게이트**(직교 앵커), AI 리뷰는 2차 확인. 계약테스트가 두꺼우면 AI 라운드를 줄인다.
- 슬라이스가 작고 경계가 명확할수록(직교 잘 됨) → 빠른 모델 1패스로 족함.

## 티어 (슬라이스/컴포넌트마다 판정)

| 티어 | 대상 | 검증 | 모델 |
|------|------|------|------|
| **T0 trivial** | 배선/glue/타입/상수, 자명한 코드 | tsc + 테스트만. AI 리뷰 0 | — |
| **T1 standard** (기본) | 일반 로직(비보안). 스킬 어댑터 대부분, 포맷·파싱, 배선 | **빠른 모델 1패스** + 테스트. "진짜버그 없음"에서 정지 | **Haiku 서브에이전트**(기본·신뢰·빠름·z.ai무관) 또는 glm-4.6(8초) |
| **T2 critical** | **경로격리·외부연결(네트워크/subprocess)·승인/권한 게이트·인증/비밀** | 2-AI 2-clean(codex + GLM-5.1 각 R1·R2 NONE) | codex(gpt-5.5) + GLM-5.1 |

## 정지 규칙 (핵심 가속)
- **"진짜 reachable 버그 없음"에서 정지.** wording/테스트-나열/정규화-대칭 같은 floor 지적은 **즉시 bound**(추가 라운드 금지). cf [[feedback_bound_threat_model_in_review_prompts]].
- T1은 **1패스로 충분**(R2/2-clean 안 함). 버그 나오면 고치고 1패스 재확인까지만.
- 위협경계 항상 명시(OOS 박아 무한 하드닝 차단).

## z.ai를 critical path에서 제거
- GLM-5.1(느림·큰 리뷰 행)은 **T2에서만**. z.ai 불안정 시 T2는 **codex 2-clean + 테스트로 커밋(B 패턴), GLM 보류**(복구 후 보강). 절대 z.ai 기다리며 블록 안 함.
- T1의 빠른 모델 = z.ai 의존 없는 **Haiku 서브에이전트** 우선(opencode glm-4.6은 대안).

## 라이브 검증
- LLM 도구루프/통합 = **헤드리스 스모크**(node, GUI 불요) 우선. Playwright/앱 GUI는 그래픽 복구 후.

## 티어 판정 가이드(빠른 분류)
- 외부 입력을 신뢰경계 넘어 처리? 경로/명령/네트워크/권한? → **T2**
- 순수 로직·포맷·이미 검증된 패턴 재사용? → **T1**
- 선언/배선/타입? → **T0**

## 적용 예 (이번 백로그 재분류)
- MCP **executor**(transport 주입, 로직) = T1+ (codex 1~2패스, GLM 보류)
- MCP **stdio transport 어댑터**(실 subprocess/JSON-RPC framing) = **T2**
- obsidian/github 경로격리·외부 = 이미 T2 처리됨(codex 2-clean 완료, GLM 보류)
- entry 배선 = T0~T1
