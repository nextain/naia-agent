# naia-agent

4-repo Naia 생태계의 **허브 런타임** 패키지.

| 레포 | 역할 |
|---|---|
| `naia-os` | Host (Tauri shell + 3D avatar + OS 이미지) |
| **`naia-agent`** (이 레포) | Runtime 엔진 + 공개 인터페이스 SoT |
| `naia-adk` | 워크스페이스 포맷 + 스킬 표준 (`@naia-adk/skill-spec`) |
| `naia-memory` | MemoryProvider 레퍼런스 구현 |

원칙: **Interfaces, not dependencies** — 공개 인터페이스로만 결합, 런타임 결합 금지. 호스트가 구현체 주입.

## 경계 원칙 (Boundary Principle)

**판단 기준 (2가지 모두 YES여야 naia-agent에 속함):**
1. "CLI로 단독 실행할 때 의미있는가?"
2. "LLM이 관리하고 호출하는가? (또는 LLM 루프에 직접 필요한가?)"

| 영역 | 속하는 것 | 메커니즘 |
|------|----------|----------|
| **naia-agent core** | LLM 루프, tool executor, memory, approval flow, skill proxy 라우팅 | 직접 구현 |
| **naia-os → 주입** | UI 스킬 (BGM, VRM, 배경, 패널, Omni, Live 등), 시스템프롬프트, UI 훅 | `skill_inject` / `panel_skills` IPC |
| **naia-adk → 확장** | 도메인 스킬, 유저 페르소나, 커스텀 훅 | `--skills-dir` (prompt.md, hooks.yaml) |

**절대 금지:**
- naia-os 전용 인프라(HTTP 서버, WebRTC, 오디오 재생 등)를 naia-agent에 내장하는 것
- naia-omni 내부 구현(LiveKit 등)을 naia-os / naia-agent 코드·문서에 노출하는 것 — naia-omni는 OpenAI Realtime API 인터페이스만 외부 노출
- "나중에 주입으로 바꾸자"며 일단 하드코딩하는 것

### Direct-mode provider resolution priority (first match wins)

`pnpm naia-agent "…"` (manifest 없음) 의 자동 해석 우선순위:

1. `ANTHROPIC_API_KEY` (+ optional `ANTHROPIC_BASE_URL`) → Anthropic 직접
2. `OPENAI_API_KEY` + `OPENAI_BASE_URL` → OpenAI-compat (generic)
3. `GLM_API_KEY` → zai/Zhipu GLM (단축, endpoint 자동)
4. `VERTEX_PROJECT_ID` + `VERTEX_REGION` → Anthropic on Vertex AI
5. (none) → ERROR exit 3, no mock fallback (Slice 3-XR-A removed the mock; `bin/naia-agent.ts buildLLMClient` returns `null` → caller exits 3 with actionable hint)

### Service-mode (`--service <manifest>`) backend enum

`*.service.json` 의 `llm.backend` 값:

| backend | 의미 | 키 요구 |
|---|---|---|
| `openai-compatible` | OpenAI-compat HTTP (GLM / vLLM / OpenRouter / ollama …) | host env (manifest 에 키 금지) |
| `anthropic` | Anthropic 직접 | `ANTHROPIC_API_KEY` host env |
